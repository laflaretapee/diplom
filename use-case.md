# Japonica — Use Cases & Демо-сценарии

> Система управления сетью суши-ресторанов: от приёма заказов до CRM франчайзинга.

---

## Что показывать клиенту

### 1. Авторизация и ролевая система

**Сценарий:** Войдите под разными ролями и покажите, что интерфейс меняется.

| Роль | Доступ |
|---|---|
| `super_admin` | Всё: пользователи, франчайзи, аналитика, склад, ИИ |
| `franchisee` | Свои точки, ИИ-ассистент, аналитика своей сети |
| `point_manager` | Заказы, склад своей точки |
| `staff` | Только очередь заказов |

**Что говорить:** *«Система знает кто вы и показывает только то, что нужно. Менеджер точки не видит другие точки, франчайзи видит только свои. Никаких избыточных прав.»*

---

### 2. Очередь заказов (реальное время)

**Маршрут:** `/orders/queue`

**Сценарий:**
1. Открыть две вкладки браузера
2. В одной вкладке создать заказ через API: `POST /api/v1/orders`
3. В другой — наблюдать, как карточка появляется в колонке «Новый» **без обновления страницы**
4. Перетащить карточку в «В работе» → «Готов» → «Выдан»

**Что говорить:** *«WebSocket-подключение держит очередь живой. Повар видит новый заказ мгновенно, без F5. При потере соединения — автоматический fallback на polling.»*

---

### 3. История заказов с фильтрами

**Маршрут:** `/orders/history`

**Сценарий:**
1. Открыть страницу истории
2. Отфильтровать по точке, статусу, диапазону дат
3. Показать цветовые бейджи статусов

**Что говорить:** *«Вся история сохраняется. Можно найти любой заказ за любой период — по точке, статусу, дате.»*

---

### 4. Склад и тех.карты

**Маршрут:** `/warehouse`

**Сценарий:**
1. Показать список ингредиентов с остатками
2. Добавить приход от поставщика (`POST /stock/supply`)
3. Создать заказ → показать, что остатки автоматически списались (write-off по тех.карте)
4. Дождаться срабатывания low-stock уведомления (если остаток упал ниже минимума)

**Что говорить:** *«Когда заказ создаётся — система сама списывает ингредиенты по рецептуре. Если запас падает ниже нормы — автоматически уходит уведомление. Никакого ручного контроля.»*

---

### 5. Внешний API для приёма заказов (Inbound)

**Сценарий:**
```bash
curl -X POST http://localhost:18080/api/v1/inbound/orders \
  -H "X-API-Key: <ключ>" \
  -H "Content-Type: application/json" \
  -d '{"point_id": "...", "items": [...], "source_channel": "delivery_club"}'
```

**Что говорить:** *«Сторонние системы — агрегаторы доставки, сайт, мобильное приложение — могут создавать заказы через единый API. Источник канала сохраняется, видно в аналитике.»*

---

### 6. Уведомления в Telegram

**Сценарий:**
1. Менеджер точки привязывает Telegram через `/notifications/telegram/link`
2. При новом заказе на его точку — в Telegram приходит уведомление с деталями
3. При низком остатке ингредиента — отдельное уведомление

**Что говорить:** *«Менеджер не обязан сидеть за компьютером. Критичные события приходят в Telegram — заказы, low-stock, любые важные изменения.»*

---

### 7. CRM франчайзинга (воронка)

**Маршрут:** `/franchisee` (только `super_admin`)

**Сценарий:**
1. Показать канбан-доску из 8 стадий: Лид → Переговоры → Договор → Обучение → Настройка → Открыт → Активен → Закрыт
2. Кликнуть на карточку франчайзи — открывается боковая панель (без перехода на другую страницу)
3. В панели показать:
   - **Обзор**: контакты, прогресс чеклиста, кнопка «Следующая стадия»
   - **Чеклист**: задачи по стадиям (добавить, выполнить, пропустить)
   - **История**: заметки менеджера с хронологией
   - **Точки**: привязать существующую точку или создать новую
4. Нажать «Добавить франчайзи» из нужной колонки — форма уже предзаполнена стадией

**Что говорить:** *«Вся воронка франчайзинга в одном экране. Видно где находится каждый кандидат, какие задачи не выполнены, какие точки уже открыты. Боковая панель не уводит с доски.»*

---

### 8. ИИ-ассистент

**Маршрут:** `/ai-assistant` (только `super_admin` и `franchisee`)

**Сценарий:**
1. Показать левую панель с быстрыми подсказками и готовыми сценариями
2. Задать вопрос: *«Какие аномалии требуют внимания на этой неделе?»*
3. Показать ответ с доказательствами и подсказками для следующих вопросов
4. Переключить контекст через селект точки

**Что говорить:** *«Аналитический ассистент работает в контексте вашей сети. Он знает данные по заказам, складу, выручке. Отвечает на бизнес-вопросы, а не только на технические.»*

---

### 9. Аналитика

**API-эндпоинты:**
- `GET /analytics/revenue` — выручка по периоду и точке
- `GET /analytics/dishes` — топ блюд по количеству и выручке
- `GET /analytics/channels` — распределение по каналам продаж
- `GET /analytics/summary` — сводка по всей сети

**Что говорить:** *«Данные агрегируются в реальном времени. Без выгрузок в Excel — любой период, любая точка, любой срез.»*

---

### 10. Безопасность и надёжность

**Что показать:**
- Refresh-токен в httpOnly cookie (не виден в JS, защищён от XSS)
- CSRF-защита на `/refresh` и `/logout`
- Rate limiting на `/login` и `/refresh`
- Security headers: `X-Frame-Options`, `X-Content-Type-Options`, HSTS
- Tenant scoping: 403 при попытке обратиться к чужой точке
- Celery worker с retry-логикой для уведомлений

**Что говорить:** *«Аутентификация следует best practices: токены в cookie, CSRF-защита, rate limiting на входе. Сотрудник физически не может прочитать данные другой точки.»*

## Если клиент спросит: «А что ты так долго?»

### Честный ответ (техническая аудитория)

*«Здесь не один модуль — их десять, и каждый сделан правильно, а не по-быстрому:*

- *Аутентификация — JWT + refresh rotation + CSRF + rate limit + bcrypt (решили конфликт библиотек)*
- *Склад — не просто таблица, а автоматическое списание по тех.картам при каждом заказе*
- *WebSocket — реальное время с fallback polling и keep-alive пингом*
- *Уведомления — Celery-очередь с retry, Telegram-интеграция, low-stock триггер*
- *CRM франчайзинга — 8-стадийная воронка, чеклисты, история, привязка точек*
- *ИИ-ассистент — с провайдером, контекстом точки, резервным режимом*
- *5 миграций Alembic, Docker Compose со всеми сервисами, security headers*

*Это продукт, а не учебный пример."*

### Ответ для бизнес-аудитории

*«Потому что сделано на вырост. Большинство "быстрых" решений — это прототипы, которые нужно переписывать как только появится второй пользователь или вторая точка. Здесь мультитенантность, разграничение прав, очереди задач и реальное время заложены с первого дня. Добавить третью точку или нового франчайзи — это 2 минуты, не неделя переработки.»*

### Ответ для преподавателя / комиссии

*«Архитектурные решения обоснованы требованиями: модульный монолит FastAPI выбран вместо микросервисов из-за границы контекста без inter-service RPC. SQLAlchemy 2.0 async — для I/O-bound операций под нагрузкой. Celery + Redis — для надёжной доставки уведомлений с retry. WebSocket с fallback polling — для UX без деградации при нестабильном соединении. Каждое решение объяснимо, а не "так было в туториале".»*

## Архитектура за 30 секунд

```
Browser ─── Nginx :18080 ─┬─ /api/v1/* ──► FastAPI :18000
                           └─ /*        ──► Vite SPA :15173

FastAPI ──► PostgreSQL :5433  (SQLAlchemy 2.0 async + Alembic)
FastAPI ──► Redis :6380       (Celery broker + Telegram linking)
Worker  ──► Redis :6380       (Celery worker: notifications, scheduled tasks)
FastAPI ──►  WebSocket         (real-time order queue per point)
```

**Стек:** Python 3.12 · FastAPI · SQLAlchemy 2.0 · Alembic · Celery · Redis · React 18 · TypeScript · Vite · Ant Design 5 · TanStack Query · Zustand · Docker Compose
